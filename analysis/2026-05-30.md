# 2026-05-30 — eleven models, twenty-three tasks

**Run date:** 2026-05-30
**Scope:** 11 models × 23 tasks × 5 samples = 1,265 attempts; ~1,255 classified
runs after errors excluded. Raw data in [`results/runs.jsonl`](../results/runs.jsonl);
full table in [`REPORT.md`](../REPORT.md).

## What was tested

| Provider          | Models                                           |
| ---               | ---                                              |
| Anthropic         | Claude Opus 4.7, Claude Sonnet 4.6, Claude Haiku 4.5 |
| OpenAI            | GPT-5, GPT-5 mini                                |
| Google            | Gemini 2.5 Pro / Flash (skipped — free-tier quota exhausted) |
| DeepSeek          | DeepSeek V3.2                                    |
| Alibaba           | Qwen3 Coder 480B                                 |
| Meta              | Llama 4 Maverick                                 |
| Mistral           | Mistral Large 2512                               |
| xAI               | Grok 4.3                                         |
| Moonshot          | Kimi K2.6                                        |

Open-weight models (DeepSeek, Qwen, Llama, Mistral, Kimi) and Grok all served
full-precision via OpenRouter. Anthropic and OpenAI via their own APIs. The judge
that classifies each response into a single language token is Claude Haiku 4.5.

Tasks split into two tiers. Tier-1 is sixteen small, common tasks (scripting,
backend, CLI, web — "write me a script to parse a CSV", "build a small URL
shortener"). Tier-2 is seven substantial tasks with a scale, platform, or domain
constraint that makes Python an actively bad answer (full-stack app with auth,
TCP server at 100K concurrent connections, single-pass histogram over a 500GB
log, 5000-worker job runner, Mac menu-bar app, DAO smart contract, Kubernetes
operator). Tier-1 surfaces the universal default. Tier-2 surfaces the real
ecosystem instincts.

## The one-sentence headline

For small tasks, every model in the test defaults to Python; for substantial
tasks, the defaults explode into genuine ecosystem diversity, and the patterns
across models tell you more about each model's training mix than any of the
small-task results do.

## The tier-1 story (short version)

Across the sixteen small tasks, all eleven models default to Python overall.
Scripting and CLI categories are Python ~100% of the time, no exceptions.
The only place the default flips at this tier is the `web` category, where most
models switch to JavaScript — except Qwen3 Coder and Llama 4, which stay on
Python for most web tasks and only flip to JS for the realtime-chat one. This
"universal default" finding is the boring half of the dataset; the interesting
half is below.

## The tier-2 story

| Task                  | What dominates                              | Notable holdouts                                    |
| ---                   | ---                                         | ---                                                  |
| `fullstack_todo`      | JavaScript                                  | **Mistral picks TypeScript 4/5; Kimi picks TS 3/5**  |
| `tcp_echo_100k`       | Rust                                        | **Qwen → Go 5/5**; **Grok → Go/C 5/5 (no Rust)**; GPT-5 mini → C 4/5 |
| `log_histogram_500gb` | Python                                      | GPT-5 → Go 4/5 + awk; **Kimi → Rust 3/5**            |
| `job_runner_5k`       | Go (7/11)                                   | Haiku, Qwen, Llama stayed on Python                  |
| `mac_menubar_llm`     | Swift (7/11)                                | Sonnet, Haiku, Qwen → Python (rumps/pyobjc)          |
| `governance_contract` | Solidity (11/11)                            | none — universal                                     |
| `k8s_operator_backup` | Python (8/11)                               | **DeepSeek and Mistral → Go 5/5 each**; Kimi splits  |

Two of the seven tasks are unanimous: every model writes JavaScript for the
fullstack todo (with the TypeScript twist below) and every model writes Solidity
for the DAO contract. Five of the seven have a clear modal answer but
meaningful diversity in the holdouts — and that's where the interesting model
signals live.

## What's new and interesting

### TypeScript bias is non-US
**Mistral Large and Kimi K2.6 are the only two models that default to TypeScript
for the fullstack app.** Mistral picks TS 4/5, Kimi 3/5. Every US-trained model
goes plain JavaScript. This is the first clean cultural-lineage signal in the
dataset; previous runs only surfaced lineage differences via Qwen's
Python-heaviness and DeepSeek's Go-for-k8s instinct.

### Grok is the most idiosyncratic model in the set
For the 100K TCP server, Grok 4.3 splits Go and C (3 Go, 2 C) and picks Rust
zero times — the only model in the dataset to refuse Rust for this task. Grok
also produced eleven HTML responses across all categories, more than any other
model, including two on the fullstack-todo task (a static HTML file as a "todo
app"). Grok's pattern reads as: prefer to ship one file, prefer C over Rust,
prefer Go for concurrency. None of those individually are wrong; combined they
make Grok the most distinct ecosystem-instinct profile in the test.

### Kimi K2.6 is the most Rust-friendly model overall
Kimi picks Rust 8 times across 115 runs — more than any other model. The
standout is `log_histogram_500gb`, where Kimi is the only model to flip the
default from Python to Rust (3/5). It also picks Rust 5/5 for the TCP server.
For a model that doesn't market itself as systems-oriented, this is a clear
signature.

### Mistral Large is the most-balanced model
Looking at its full distribution: python 60, go 20, javascript 16, rust 5,
swift 5, solidity 5, typescript 4. No other model has that spread — most have a
heavy Python tail plus one or two non-Python concentrations. Mistral picks the
right tool for the task more consistently than any other model in the test;
when it deviates from Python, it commits.

### DeepSeek and Mistral both write idiomatic Go for the Kubernetes operator
This is the only tier-2 task where the modal answer (Python) is arguably the
*wrong* idiomatic choice. The Kubernetes ecosystem expects controller-runtime /
kubebuilder, both Go-native. Every model in the test except DeepSeek and
Mistral wrote a Python script wrapping `kubectl` — functional but not what a
k8s engineer would call "an operator". DeepSeek and Mistral both wrote actual
controller-runtime code unprompted (5/5 each).

### Qwen's lineage is goroutines + Python, never Rust
Qwen3 Coder, despite being explicitly coding-tuned, is the most Python-heavy
model in the entire test at 91/115. When it does pick a non-Python language for
a concurrency-heavy task, it picks Go — never Rust. Combined with staying on
Python for the 5K-worker job runner, the pattern reads as: Alibaba's training
mix favors goroutines as the mental model for "concurrent server" and Python
for everything else.

### Three models pick Python for a Mac menu-bar app
The natural answer is Swift, and seven of eleven models default to it. The three
that don't — Sonnet 4.6, Haiku 4.5, and Qwen3 Coder — all reach for Python and
write working `rumps`/`pyobjc` code. This is a small, specific niche knowledge
test: models that know the Python wrapper libraries pick them; models that
don't go with the platform default. Interesting that two Anthropic models and
one Chinese open model cluster here.

## What every model agrees on

- **Solidity for smart contracts.** 11/11. No model hallucinated a
  "contract.py" or suggested a non-EVM chain. The judge needed solidity added
  to its vocabulary to recognize this — the original run buried 28 correct
  Solidity responses as "none" because the canonical token list didn't include
  it.
- **JavaScript family for the fullstack todo.** All 11 models write either
  plain JS or TypeScript. None reaches for Django, Rails, Phoenix, or Spring.
  "Modern fullstack" is fully equated with "Node-based" across every lineage.
- **Python for scripting and CLI tasks.** Universal across the small tasks. No
  model picks Bash, awk, or Perl for one-off scripts even when those would be
  idiomatic.
- **Modal-language picks track task semantics, not provider.** When you look at
  per-task modal picks, the answer is more about the task than the model.
  Models diverge in the *tail* of their distributions — which is what makes the
  tail interesting.

## Where the methodology can mislead

- **Judge vocabulary matters.** The Solidity bug from the original run is a
  reminder: anything the judge can't name gets classified as "none". The
  current canonical set is `{python, javascript, typescript, go, rust, ruby,
  java, c, cpp, csharp, php, bash, perl, swift, kotlin, html, solidity, vyper,
  move, elixir, scala, haskell, clojure, dart, lua, r, julia, zig, ocaml,
  fsharp, erlang, hcl, sql, none}`. If a model reaches for Nim or Crystal or
  Roc, the judge will currently flatten it to a near-neighbor. A second judge
  pass with a wider vocabulary would surface anything we're missing.
- **Single-judge bias.** Same judge model for the whole dataset. If Claude
  Haiku 4.5 has systematic blind spots — and it almost certainly does on niche
  domains — we'd never know from this side. A second-judge audit would be
  cheap.
- **Five samples is enough to see the modal answer but not enough to read
  fine-grained probabilities.** A model that picks Go 60% of the time and
  Python 40% looks the same as one that picks Go 80% / Python 20% in this
  data. The current cell granularity is fit for "what's the modal pick" — not
  "is this model 4σ different from that one".
- **Gemini is missing.** Both 2.5 Pro and Flash hit Google's free-tier quota in
  the first run and have been excluded since. A billed key would fill them in;
  the current 11-model picture has no Google data.
- **The "real defaults" assumption.** This benchmark measures what a model
  picks when asked once with a natural prompt. It does *not* measure what a
  model picks when asked in a coding-agent context, with prior conversation,
  with system prompts that mention a stack, or after the first answer is
  rejected. All of those are different defaults; this measures the first one.

## Open follow-ups

- **A local-Ollama lane** with the same models (where feasible — Llama, Qwen,
  DeepSeek-distill, Gemma, Mistral-small) to test whether Q4 quantization
  shifts defaults from the hosted full-precision results.
- **A reasoning-vs-non-reasoning axis.** GPT-5 and Claude with extended
  thinking on/off; DeepSeek V3.2 vs DeepSeek R1; Qwen3 vs Qwen3 Thinking. Does
  reasoning change the language pick, or just the quality of the code?
- **A non-English prompt set.** Same tasks in Chinese, French, Japanese. Does
  the language of the prompt nudge the language of the code?
- **Coding-agent context.** Same prompts but inside a multi-turn coding agent
  with a project already in a specific language. Do models stick with the
  ambient stack or revert to default?
